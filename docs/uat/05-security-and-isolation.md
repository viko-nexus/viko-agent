# 05 — Security & Isolation

`viko-nexus/viko-agent` is a public GitHub repository, which means the entire
isolation design — the boot-time guard, the network topology, the mount
allowlist, secrets handling — is readable by anyone. That makes this the
highest-stakes test area in the plan: these cases exist to prove the
enforcement actually holds at runtime against a project container or an
unauthenticated network caller, not merely that the design document reads
correctly. Several cases are deliberately adversarial (path traversal, forged
tokens, malformed input) and must be run in a disposable/test environment,
never production.

| ID | Precondition | Steps | Expected Result |
|---|---|---|---|
| SEC-01 | A throwaway/test project container can be spawned with a deliberately broken mount or missing isolation prerequisite (e.g. missing relay token, wrong SSH alias); `VIKO_ISOLATION_GUARD=enforce` (default). | 1. Spawn the misconfigured container.<br>2. Observe container status and logs. | The container goes permanently inert at boot: `docker ps` shows it unhealthy/exited, a tombstone file is written, a best-effort alert fires, and the gateway process never starts. Failure is loud and visible, not silent. |
| SEC-02 | Same misconfiguration as SEC-01, `VIKO_ISOLATION_GUARD=warn`. | 1. Spawn the same misconfigured container with warn mode set.<br>2. Observe container status and logs. | The guard logs the same findings but does NOT block startup — the gateway starts normally. Confirm this documents the operational difference from enforce mode (for ops runbooks, not for use in prod). |
| SEC-03 | Same misconfiguration, `VIKO_ISOLATION_GUARD=off`. | 1. Spawn the container with the guard disabled.<br>2. Check that no isolation checks ran at all (absence of guard log lines).<br>3. Separately, grep the production `.env`/compose config to confirm `VIKO_ISOLATION_GUARD=off` is never set there. | Guard is skipped entirely — no checks run, no logs. Confirm production config never sets this value (it should be `enforce` or unset/default). |
| SEC-04 | A running project container (`viko-{slug}`), aware of at least one sibling project's slug. | 1. `docker exec` into the project container.<br>2. Attempt to read a sibling project's directory via a symlink or relative path escape derived from the slug (e.g. `../other-slug/`, or a crafted symlink inside the container's writable area pointing outside its mount). | Read is denied or the path does not exist inside the container's filesystem view — the container's mount namespace has no visibility into other projects' directories regardless of path tricks. |
| SEC-05 | Two onboarded projects A and B; both project networks can reach the admin bridge (hub-spoke topology). | 1. From project A's container, `curl` the admin bridge's scoped endpoints (e.g. `/messages`, `/group/{B's jid}/participants`) using either A's own token against B's JID, or no token at all. | `403` in all cases (`cross_project_send_blocked`, `cross_project_read_blocked`, or `unknown_relay_token` as applicable — see `02-whatsapp-bridge-and-routing.md` for the detailed matrix). Confirm this is the real boundary: L3 network reachability to the hub exists, but the token/scope check denies the read. |
| SEC-06 | A project container is running. | 1. `docker exec viko-{slug} printenv \| grep -i github_token`. | No `GITHUB_TOKEN` present in the project container's environment. |
| SEC-07 | A project container is running. | 1. `docker inspect viko-{slug}` and check mounted volumes/binds. | No `/var/run/docker.sock` (or any docker socket) is mounted into the project container. |
| SEC-08 | A project container is running. | 1. `docker exec viko-{slug} ls /opt/...` (wherever `mcp-servers/` would be mounted in the admin container) or otherwise attempt to locate `projects-gateway.py` inside the project container. | The path does not exist / `mcp-servers/` is entirely absent from the project container's filesystem. |
| SEC-09 | A project container is running. | 1. `docker inspect viko-{slug} --format '{{json .Mounts}}'` and enumerate every bind mount. | Mounts are limited to: the project's own code (`$VIKO_PROJECTS_ROOT/{slug}`, rw) plus the documented read-only allowlist (`AGENTS.md`, `rules/`, `soul/`, `skills/`, `projects/{slug}`). No `.env`, no other project's `data/`, no unrelated repo paths appear. |
| SEC-10 | Admin dashboard reachable at `:9119`, `HERMES_DASHBOARD_INSECURE=false` (default/expected). | 1. `curl -i http://<host>:9119/` (or open in a browser) with no credentials. | `401` (or a login prompt/redirect) — the dashboard content itself is not returned. |
| SEC-11 | Same as SEC-10. | 1. Attempt login with an incorrect username/password. | Login is rejected; no session/dashboard access is granted. |
| SEC-12 | Same as SEC-10; correct credentials known (from `.env`). | 1. Attempt login with the correct username/password (or password_hash+secret combo). | Access is granted; dashboard loads normally. |
| SEC-13 | A disposable, non-production test environment (never a live VPS). | 1. Set `HERMES_DASHBOARD_INSECURE=true` in the test env's `.env`.<br>2. Restart the dashboard/admin container.<br>3. From another host/container on the same docker network, curl the dashboard with no credentials. | Dashboard is reachable network-wide with no auth — confirms the documented warning is accurate. **Do NOT set this in production**; tear down the test env after this case. |
| SEC-14 | A new project spawn is about to run (`spawn-hermes.py`). | 1. Immediately as `docker run` executes, capture `ps aux` on the host (or `docker top`) and separately run `docker inspect --format '{{json .Config.Env}}' viko-{slug}` on the freshly-started container. | `ps aux`/argv output does NOT show `OPENAI_API_KEY`, `NINEROUTER_KEY`, or `HERMES_RELAY_TOKEN` in plaintext (they were passed via a mode-0600 `--env-file`, deleted right after `docker run` completes). **Acknowledge**: `docker inspect` on the running container DOES show these env vars — this is a documented limitation of Docker itself, not a bug in this design; do not fail the test case on that basis. |
| SEC-15 | `spawn-hermes.py` accessible for a direct/manual test invocation. | 1. Invoke `spawn-hermes.py` (or the onboarding path that calls it) with a slug containing `../` or `/`, e.g. `../../etc` or `foo/bar`. | Rejected before any side effect — no directory created under `$VIKO_PROJECTS_ROOT`, no container spawned, no `routing.json` entry written. Validation against `[a-z0-9][a-z0-9-]{0,38}` fails closed prior to any filesystem/docker action. |
| SEC-16 | Same as SEC-15. | 1. Invoke the same path with a malformed `group_jid`, e.g. `not-a-jid`, `12345` (missing `@g.us`/`@s.whatsapp.net`), or one with non-digit characters. | Rejected before any side effect, same as SEC-15 — no partial provisioning occurs. |
| SEC-17 | Onboarding a project with GitHub SSH access. | 1. During onboarding, inspect how `known_hosts` is seeded for `github.com` for the new project's SSH config. | The seeded host key matches Anthropic-independent, publicly documented GitHub SSH host key fingerprints (pinned in the onboarding code/config), NOT the output of a live `ssh-keyscan github.com` run at onboarding time — confirms MITM-poisoning resistance at first contact. |
| SEC-18 | At least one onboarded project exists. | 1. On the host/VPS, check permissions: `stat -c '%a %n' data/bridge/routing.json` (expect 600) and its directory (expect 700).<br>2. Check `/home/deploy/{slug}/config/project.json` (expect 600).<br>3. Check the per-project SSH private key file (expect 600). | All three match spec exactly: `routing.json` is mode 600 inside a mode 700 directory; `project.json` is mode 600; the per-project SSH key is mode 600. None are group- or world-readable. |
