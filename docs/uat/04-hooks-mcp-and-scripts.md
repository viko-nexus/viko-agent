# 04 — Hooks, MCP & Scripts

Outside the core agent loop, the system leans on event hooks (`hooks/`), MCP
tool servers (`mcp-servers/`), and standalone cron/ops scripts (`scripts/`)
to handle startup notification, commitment tracking, media delivery,
cross-project SSH (admin only), and fleet housekeeping (provisioning,
pruning, idempotent init). These are the pieces most likely to silently
break without anyone noticing — cron jobs run unattended and MCP tools fail
open/closed depending on mounting — so this file verifies each one's
observable effect, not just that the script exists.

| ID | Precondition | Steps | Expected Result |
|---|---|---|---|
| OPS-01 | `WHATSAPP_HOME_CHANNEL` is set to a valid group JID the admin bridge can reach. | 1. Restart the admin container (`docker compose --profile full up -d --force-recreate hermes`).<br>2. Watch the home channel. | Within ~60s, the home channel receives "Online lagi! Siap nerima perintah 🟢" (the `viko-startup` hook retries every 5s, up to 12x, until bridge readiness). |
| OPS-02 | `WHATSAPP_HOME_CHANNEL` is unset/empty. | 1. Restart the admin container with the var unset.<br>2. Check `docker logs viko-hermes` for errors. | No error is logged for the missing home channel; the startup hook is a clean no-op (no retry spam, no crash). |
| OPS-03 | Owner has an active session in a project (VIKO_PROJECT_SLUG set for that container). | 1. As owner, send a message implying a future action, e.g. "aku cek lagi besok" or "I'll follow up on this tomorrow".<br>2. Inspect that project's `commitments.json`. | A new entry is appended with the detected commitment phrase and a sensible due date (default: next day) derived from the phrase. |
| OPS-04 | A due (or past-due) commitment exists in a project's `commitments.json`, and `deliver_commitments.py`'s cron is installed (see OPS-14). | 1. Wait for the 15-minute cron tick (or invoke `deliver_commitments.py` manually for the test).<br>2. Check the correct WA group (resolved via `routing.json` slug→JID). | The commitment message is POSTed to `localhost:3000/send` and delivered to the correct project's WA group at or after its due time. |
| OPS-05 | Owner asks the agent to produce a deliverable (e.g. a screenshot). | 1. Ask Viko to take a screenshot.<br>2. Contrive (or observe naturally) a reply that claims the file was sent but is missing a working `MEDIA:` tag. | The `viko-media-autosend` hook detects the claim, resolves the most recent deliverable (screenshot/pdf/doc/video), and sends it via `/send-media` as a safety net — the file arrives in the chat despite the missing tag. |
| OPS-06 | Same deliverable already sent once via OPS-05. | 1. Ask for the same thing again immediately (same file, same chat). | No duplicate send occurs — de-dup at the bridge (same chat + filename + size) suppresses the repeat. |
| OPS-07 | Owner is interacting with Hermes-Admin (or an admin-context session) with a valid target project configured for SSH. | 1. As owner, ask Viko to check logs on `<project>`'s prod server (something that would invoke `ssh_exec`). | `ssh_exec(project, command)` runs via `mcp-servers/projects-gateway.py`, executing `ssh -F /opt/data/.ssh/config -o ConnectTimeout=15 <ssh_host> <command>` and returning stdout+stderr to the agent, which relays it to the owner. |
| OPS-08 | A project-scoped Hermes container (`viko-{slug}`) is running. | 1. From inside that project's container/session, attempt to invoke any `ssh_exec`-style tool targeting a DIFFERENT project. | No such tool is available at all — `mcp-servers/` (the cross-project SSH gateway) is not mounted into project containers. Confirm via the agent's tool list / an explicit "tool not found" style failure, not a permission-denied at execution time. |
| OPS-09 | `viko-tools.py`'s `generate_chart` is available in the session. | 1. Ask Viko for a bar chart from a small dataset (e.g. "chart these 3 numbers: 10, 20, 15"). | A matplotlib-rendered PNG is generated and delivered to WhatsApp via `/send-media`. |
| OPS-10 | A known local file path is available to the session (e.g. a generated report). | 1. Ask Viko to "send this file" by path, specifying media type and caption. | The file arrives in WhatsApp with the correct `media_type` and the specified caption, sent via `send_wa_file` → `/send-media`. |
| OPS-11 | A new project slug and GitHub URL ready for onboarding; note the current highest allocated port in `data/bridge/routing.json`. | 1. Run `add-project.py` end-to-end for the new project (or via the onboarding wizard).<br>2. Check the allocated port in `routing.json`.<br>3. Interrupt a second `spawn-hermes.py` run mid-write if feasible (e.g. kill during the routing.json update) and inspect the file. | Port allocation increments correctly from the last-used port (starting at 8101). `routing.json` update is atomic — even if a run is interrupted mid-write, the file remains valid JSON (no partial/corrupt content), because the update writes to a temp file and renames. |
| OPS-12 | A phone number not yet in `WHATSAPP_ALLOWED_USERS`. | 1. Run `allow-member.py <number>`.<br>2. Immediately attempt a DM from that number to Admin. <br>3. Run the restart command `allow-member.py` printed.<br>4. Retry the DM. | After step 1 alone, DM access does NOT yet work (the `.env` change requires a Hermes restart to take effect). After step 3 (restart), the DM succeeds. Re-running `allow-member.py` for the same number is idempotent (no duplicate entries in `.env`). |
| OPS-13 | A project session idle beyond the configured threshold (default 6h via `--idle-hours`); `dream_sessions.py` reachable (9router `viko-chat`/haiku model available). | 1. Leave (or simulate) a session idle past the threshold.<br>2. Run `prune-idle-sessions.py` (or wait for its hourly cron).<br>3. Check `sessions.json.bak` and the project's `context.md`. | The idle session is removed from the live session store; `sessions.json.bak` is written as a backup before deletion; a dreamed (summarized) block is appended to that project's `context.md` before the session is deleted. |
| OPS-14 | Deploy user's crontab does not yet have the two jobs installed. | 1. Run `setup-cron.sh`.<br>2. Inspect crontab (`crontab -l`) and note the two entries.<br>3. Run `setup-cron.sh` again.<br>4. Inspect crontab again. | After the second run, the crontab still has exactly one entry for each job (no duplicates) — the installer is idempotent. |
| OPS-15 | 9router already has `viko-chat`/`viko-code` combos from a prior run. | 1. Run `init-9router.py`.<br>2. Inspect 9router's combo list/SQLite state before and after. | Second (and subsequent) runs make no changes — existing combo definitions are updated in place if needed, never duplicated. |
| OPS-16 | `data/hermes/config.yaml` already has the required overrides applied from a prior run. | 1. Run `init-hermes-config.py`.<br>2. Diff `config.yaml` before/after the second run. | Second run produces no diff — overrides are applied idempotently (re-applying an already-present override is a no-op). |
