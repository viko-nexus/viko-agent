# 06 — Deployment & Versioning

The CI/CD pipeline (`.github/workflows/deploy.yml`) rebuilds and redeploys
the admin Hermes image on every qualifying push to `main`, but — critically —
it does NOT touch already-running per-project containers, and 9router floats
on an unpinned `:latest` tag. These cases verify the pipeline's gating logic
(quality → build → deploy → release) actually behaves as designed, and make
explicit the two gaps operators must know about: the manual per-project
respawn requirement, and 9router's lack of version pinning/rollback. This
session's real version bump (2026-07-10, `viko-hermes` recreated, per-project
containers left on the prior image digest) is used below as the reference
walkthrough — repeat the same observation steps for future bumps.

| ID | Precondition | Steps | Expected Result |
|---|---|---|---|
| DEP-01 | Write access to push to `main` (or a test fork with the workflow enabled), a trivial change ready for `Dockerfile.hermes` or `patches/`. | 1. Push a commit touching `Dockerfile.hermes` (or any file under `patches/`) to `main`.<br>2. Open the workflow run, inspect the build job's "Check if image rebuild needed" step output.<br>3. Separately, push a commit that touches neither path (e.g. a comment-only change elsewhere) and inspect the same step. | For step 1: the step reports rebuild-needed=true, and the build job actually builds and pushes `ghcr.io/viko-nexus/viko-agent:latest`. For step 3: the step reports rebuild-needed=false, and no image build/push occurs (cache/cost is preserved). |
| DEP-02 | A change limited to `docs/` or `scripts/` only (no `Dockerfile.hermes`/`patches/` touched). | 1. Push the docs/scripts-only change to `main`.<br>2. Inspect the workflow run's job list and each job's conclusion. | The `quality` job runs (ruff + `npm run check` still execute). The `build` and `deploy` jobs report skipped/no-op appropriately (build doesn't rebuild per DEP-01; deploy has nothing new to pull). |
| DEP-03 | A throwaway branch or fork — **NOT `main`**. | 1. On the throwaway branch, deliberately break a patch script's assumption (e.g. edit `patches/patch-model-router.py` to reference a Hermes source file/line that no longer exists post-upstream-changes, or otherwise make its `RUN python3 /tmp/patch-*.py` step fail).<br>2. Push and trigger the workflow (or open a PR if the workflow supports manual dispatch on the branch). | The `build` job fails loudly (non-zero exit on the broken `RUN python3 /tmp/patch-*.py` step). The `deploy` job never runs — its `needs: build` dependency blocks it. Confirm no image reaches GHCR/`:latest` and no VPS deploy is attempted from this broken build. |
| DEP-04 | A successful deploy has just completed (e.g. following the DEP-01 rebuild path all the way through). | 1. SSH the VPS.<br>2. `docker inspect viko-hermes --format '{{.Image}}'` and note the digest.<br>3. For each running `viko-hermes-{slug}` project container, run the same inspect and compare digests. | `viko-hermes` (admin) shows the NEW image digest (freshly force-recreated by the deploy job). Every `viko-hermes-{slug}` project container still shows its PREVIOUS digest, unchanged by the deploy. This is expected behavior, not a bug — document it so testers know a patch/behavior change requires a manual re-spawn (`spawn-hermes.py` or `add-project.py`'s re-onboard path) to reach live projects. |
| DEP-05 | An onboarded project container running normally; ability to induce a health-check failure in a freshly spawned replacement (e.g. point it at a broken config for the test). | 1. Note the current relay token for the project in `routing.json`.<br>2. Trigger a re-spawn via `spawn-hermes.py` for that slug, engineered so the new container fails its health check.<br>3. Observe the outcome. | The atomic re-spawn design rolls back: the OLD container (renamed `{name}-prev` during the attempt) is restored as the live container, and `routing.json` is rolled back to the OLD relay token — never left pointing at a dead/failed container. |
| DEP-06 | `viko-9router` running, `docker-compose.yml`'s 9router service on the floating `decolua/9router:latest` tag. | 1. Note the current version: `docker exec viko-9router cat /app/package.json \| grep version`.<br>2. Run `docker compose pull 9router && docker compose up -d --force-recreate 9router`.<br>3. Re-check the version and container health. | The version string in `package.json` inside the container reflects whatever was newest on Docker Hub at pull time (may or may not have changed since the last pull). Container reports healthy after recreation. Confirm testers understand there is no rollback path beyond re-pulling an older tag, if one happens to exist on Docker Hub. |
| DEP-07 | `Dockerfile.hermes`'s `ARG HERMES_COMMIT` pinned to a specific tagged upstream release. | 1. `git ls-remote --tags https://github.com/NousResearch/hermes-agent` (or check the GitHub releases API) and compare the latest published tag to the currently pinned `HERMES_COMMIT`. | This is a manual staleness check — confirm the process is documented (e.g. recommended monthly cadence) since there is no automated alert for upstream drift. Record the gap (number of releases behind) as informational, not pass/fail. |
| DEP-08 | A conventional-commit-worthy change (`feat:` or `fix:`) ready to merge to `main`, vs. a `chore:`/`docs:` change. | 1. Push a `feat:` or `fix:` commit to `main` and observe the `release` job.<br>2. Separately, push a `chore:` or `docs:`-only commit and observe the same job. | For the `feat`/`fix` push: `release` (semantic-release) creates a new semver tag and GitHub release, and re-tags the already-pushed `:latest` GHCR image with the version number (e.g. `v0.8.1`) matching the new `package.json` version. For the `chore`/`docs` push: no new release/tag is created (default conventional-commits config excludes these types from version bumps). |
| DEP-09 | Workflow supports `workflow_dispatch`; no new commits pending on `main`. | 1. Manually trigger the workflow via `workflow_dispatch` (GitHub Actions UI or `gh workflow run`) with no preceding code push. | `quality`, `build`, and `deploy` all run correctly for this ad-hoc trigger (build may no-op if no qualifying paths changed since the last image, but the jobs execute rather than being skipped entirely) — confirms manual redeploys work independent of a push event. |
