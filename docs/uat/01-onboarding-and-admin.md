# 01 — Onboarding & Admin

Hermes-Admin ("Viko") is the only agent instance with the authority to
provision new projects, and its authorization model (`admin/rules/authorization.md`)
is what stops a random WhatsApp contact from spawning containers or reading
someone else's project. These cases verify the persona boundaries hold under
real traffic and that the onboarding wizard — the only path that creates new
attack surface (containers, SSH keys, routing entries) — behaves correctly
end-to-end, including its rollback-on-cancel guarantee.

| ID | Precondition | Steps | Expected Result |
|---|---|---|---|
| ADM-01 | Owner is a member of a WhatsApp group not present in `data/bridge/routing.json`. | 1. As owner, send `hi` in the unregistered group.<br>2. Observe Admin's reply. | Admin replies with the fixed invite/onboarding line (e.g. pointing to the `onboard` command), not a command menu or free-form chat. |
| ADM-02 | Non-owner is a member of a WhatsApp group not present in `routing.json`. | 1. As a non-owner participant, send any message (e.g. `hi`, `onboard`) in the unregistered group. | No reply is sent by Admin. Bridge logs show the message was silently ignored (per authorization.md: unregistered + non-owner → no reply). |
| ADM-03 | A group JID IS present in `routing.json` (already onboarded to a project). | 1. As owner, send a message directly in that registered group (not addressed to the project agent, e.g. plain admin-style command like `status`).<br>2. As non-owner, do the same. | Neither message reaches Hermes-Admin's LLM. The bridge forwards blind to the routed Hermes-Project per routing.json; Admin never answers in a registered group under any sender, per the "no exception" rule in authorization.md. |
| ADM-04 | Non-owner has Admin's WhatsApp number/contact for DM. | 1. As non-owner, send a DM to Hermes-Admin (any content, including something that looks like a command, e.g. `onboard slug test`). | Admin replies with the fixed brief canned reply only. No command is executed, no container is touched. |
| ADM-05 | Owner has Admin's WhatsApp number/contact for DM. | 1. As owner, DM Admin: "what is Viko?" or "how do I onboard a project?" or "what did we onboard recently?" | Admin gives a genuinely helpful, on-topic answer (not the canned non-owner reply, not a refusal). |
| ADM-06 | Owner DMs or group-messages Admin in an unregistered group; a real GitHub repo URL and a reachable SSH host are available for test. | 1. Owner sends `onboard`.<br>2. Step 1 (Name): provide a project name; confirm auto-slugified value.<br>3. Step 2 (Repo): provide a single GitHub URL.<br>4. Step 3 (Server): provide host/user (omit port to confirm default 22 / `viko-exec`).<br>5. Step 4 (Members): confirm candidate list is offered; pick one or more.<br>6. Step 5 (Confirmation): review summary, reply `ya`.<br>7. Step 6 (Execution): observe pubkey output, reply `ok` once added as deploy key on GitHub, wait for SSH verification. | Wizard completes one question per turn, in order. Final handoff message is sent; `routing.json` has a new entry for the slug; `docker ps` shows a new `viko-{slug}` container; SSH verification (`docker exec ... ssh {slug}-prod "echo viko-ok"`) succeeds and is shown to the owner. |
| ADM-07 | Same as ADM-06 but no SSH host available/desired. | 1. Run wizard through Step 2 (Repo).<br>2. At Step 3 (Server), reply `skip`.<br>3. Continue through Members, Confirmation, Execution. | Project is provisioned as local-only, no SSH. The SSH verification step (`docker exec ... ssh {slug}-prod ...`) is skipped entirely — confirm no SSH attempt appears in the execution log/transcript, and the handoff message does not claim SSH was verified. |
| ADM-08 | Two real GitHub repo URLs available. | 1. Run wizard to Step 2 (Repo).<br>2. Provide `label1=url1,label2=url2` (2 entries).<br>3. Complete remaining steps. | Project is provisioned as multi-repo. `add-project.py` is invoked with `--repo-subdir` set per repo/label. Confirm both repos are cloned under the project's `repo/` tree in their respective subdirs, each labeled as provided (or auto-derived from repo name if a label was omitted on one entry — test that sub-case too). |
| ADM-09 | Group has at least one participant whose WA identity only resolves to a `lid` (not a phone number) per `GET localhost:3000/group/{group_jid}/participants`. | 1. Run wizard to Step 4 (Members).<br>2. Inspect the candidate list Admin presents. | Only participants with `resolved: true` phone numbers appear as pickable candidates. The unresolved-lid participant is never listed as an option (not even as a raw lid string) — owner may still add them via "manual number" entry only if they know the number out-of-band. |
| ADM-10 | Unregistered group or DM; complete repo URL and slug decided in advance. | 1. Owner sends `viko onboard slug myproj github https://github.com/org/myproj vps host=1.2.3.4 user=deploy members 6281234567890`. | Wizard is fully skipped — Admin proceeds straight to Step 5 (Confirmation) or Execution with all fields pre-filled from the one-liner. No individual step questions are asked for fields already supplied. |
| ADM-11 | Same as ADM-10 but the one-liner omits one required field (e.g. no `vps ...`, or no `members ...`). | 1. Owner sends `viko onboard slug myproj github https://github.com/org/myproj` (server and members omitted). | Wizard starts but skips Steps 1–2 (already filled), and prompts only for the missing steps (Server, Members) before Confirmation. |
| ADM-12 | Wizard in progress, at Step 1 (Name/slug). | 1. Owner sends `cancel`. | Wizard stops immediately at Step 1. No container, routing entry, or data dir was ever created (nothing to roll back yet) — confirm via `docker ps` and `routing.json` that no trace exists. |
| ADM-13 | Wizard in progress, at Step 2 (Repo), no repo confirmed yet. | 1. Owner sends `cancel` mid-Step-2. | Wizard stops. No `docker rm`, no routing entry, no `data/hermes-{slug}/` was created (execution had not started). Confirm no partial clone exists under `$VIKO_PROJECTS_ROOT`. |
| ADM-14 | Wizard in progress, at Step 3 (Server). | 1. Owner sends `cancel` mid-Step-3. | Same rollback guarantee as ADM-12/13 — no side effects to undo since execution hasn't run; confirm cleanly. |
| ADM-15 | Wizard in progress, at Step 4 (Members). | 1. Owner sends `cancel` mid-Step-4. | Same as above — no side effects prior to execution. |
| ADM-16 | Wizard in progress, at Step 5 (Confirmation), summary already shown. | 1. Owner sends `cancel` instead of `ya`. | Wizard stops before Execution (Step 6) ever runs. No `add-project.py` invocation occurred; no container, routing entry, or data dir exists. |
| ADM-17 | A group JID is already present in `routing.json` (previously onboarded). | 1. As owner, in that (or a different) context, initiate `onboard` targeting the same already-registered group JID without stating explicit re-onboard intent (e.g. just "onboard" or repeating the original request). | Admin responds with the pre-wizard guard message ("Project ini sudah terdaftar...") and stops. No wizard steps proceed, no `--force` execution occurs. |
| ADM-18 | Same precondition as ADM-17. | 1. As owner, explicitly state re-onboard intent (e.g. "onboard ulang project ini" / "re-onboard, force it"). | Admin proceeds through the wizard and, at Execution, invokes `add-project.py` with `--force`. The existing container/config is replaced; `routing.json` entry for the JID is updated (not duplicated). |
| ADM-19 | `WHATSAPP_OWNER_NUMBER` is unset or empty in the admin container's environment. | 1. Start/restart the admin container with the var unset.<br>2. As any sender (including what would normally be the owner's number), send any command, e.g. `onboard`. | No command is processed for any sender. `docker logs viko-hermes` shows a startup warning about the missing/empty `WHATSAPP_OWNER_NUMBER`. Confirm this in both a group and a DM context. |
| ADM-20 | A registered project exists; owner has Admin's contact. | 1. As owner, DM (or message in an unregistered context) Hermes-Admin: `viko offboard`. | Admin does NOT offboard the project itself. It replies redirecting the owner to run the offboard command inside the target project's own group/channel instead. |
| ADM-21 | Repo checked out locally. | 1. Open `CLAUDE.md` and note the documented path for the onboarding skill: `admin/skills/onboarding.md`.<br>2. Check the actual filesystem path: `admin/skills/viko/viko-onboarding/SKILL.md`. | **Doc-accuracy finding, not a runtime defect**: the two paths differ. File a doc fix so `CLAUDE.md`'s "Key Files" table points at the real path (`admin/skills/viko/viko-onboarding/SKILL.md`), since a future contributor following the doc literally will not find the file. |
